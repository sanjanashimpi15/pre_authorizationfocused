# Design Walkthrough & Justifications — Taiga Financial Compliance Engine

This document explains every architectural decision made in designing the **Taiga Financial Compliance Engine** and its precise position within the Aivana Insurance Intelligence Platform.

---

## 1. Architectural Q&A (All 12 Questions)

### Q1. What is the best production architecture?
A **parallel multi-engine orchestration model** where five validation engines run concurrently on the merged TPR + CEA input, then feed into a sequential Package Validation and Financial Risk Scoring phase. The parallel phase covers the five largest workloads (ICD, Procedure, Policy, Room Rent, Billing). Because these five are independent of each other's outputs, they can all start simultaneously. Package Validation is the only engine that must wait — it depends on both ICD and Procedure results. This design achieves sub-730ms average latency on the deterministic path.

The key architectural principle is the same as Fairway's: **facts are discovered deterministically; AI may only participate in interpretation tasks**. In Taiga's case, this means financial calculations are always deterministic, while medical terminology resolution is the only zone where AI is permitted.

---

### Q2. Should ICD coding be deterministic first?
**Yes, always.** ICD validation follows a strict three-tier cascade:

```
Tier 1: Exact deterministic lookup (ICD dictionary + alias table)
         ↓ if no match
Tier 2: Chapter lock validation (deterministic specialty-to-chapter rules)
         ↓ if diagnosis text ambiguous
Tier 3: LLM ICD Suggestion Adapter (conditional, invoked < 20% of cases)
         ↓ if confidence < 0.85
Tier 4: Human review hold
```

The reason deterministic must come first: an LLM producing an ICD code that violates a chapter lock (e.g., suggesting an `O` code for a cataract case) would be an audit failure. The chapter lock check runs deterministically before any LLM output is accepted.

Additionally, any ICD code generated by AI with confidence < 0.85 automatically blocks claim submission. No financial decision in Taiga depends on an ICD code that isn't at least 85% confident.

---

### Q3. How should AI participate?
AI is permitted **only in medical terminology normalization** tasks — never in financial calculations. The exact allowlist:

| AI Task | Trigger Condition |
| :--- | :--- |
| ICD suggestion | Diagnosis text not found in deterministic dictionary |
| Procedure normalization | Procedure name not found in synonym/mapping table |
| Drug name resolution | Generic/brand name mismatch requires pharmacological knowledge |
| Diagnosis terminology normalization | Abbreviation expansion (e.g., "STEMI" → "ST-Elevation Myocardial Infarction") |

The AI never touches:
- Room rent arithmetic
- Co-pay calculations
- Waiting period date arithmetic
- Package rate selection
- Financial risk classification
- Exclusion matching

This guarantees **100% arithmetic accuracy** and **100% financial calculation accuracy** as required.

---

### Q4. How should policy rules be represented?
**YAML-based policy files with versioned schemas.**

Four formats were evaluated:

| Format | Verdict | Reason |
| :--- | :---: | :--- |
| **YAML** | ✅ Chosen | Human-readable, git-diffable, hot-reloadable, no JVM |
| JSON | ❌ | No comments, poor readability for complex nested rules |
| Drools | ❌ | Requires JVM runtime, complex deployment, steep learning curve |
| Decision Tables (Excel/CSV) | ❌ | Cannot be version-controlled cleanly, no validation |

YAML policy files are:
- **Versioned** in source control (one file per insurer-scheme-year)
- **Hot-reloadable** (no engine restart required)
- **Auditable** (the exact YAML version used in a case is recorded in the audit log)
- **Diffable** (compliance teams can review what changed between policy years)

Rule resolution uses a priority ladder: Hospital Override > Insurer Scheme Rule > Insurer Global Rule > IRDAI Standard Rule. This means hospital-negotiated package rates apply first, IRDAI non-payable rules apply last.

---

### Q5. How should room rent calculations work?
Room rent is the most financially impactful validation in Indian hospital claims. The algorithm is **100% deterministic** and follows the IRDAI proportional deduction standard:

**Step 1**: Read the policy room cap as a percentage of Sum Insured per day.
- Normal Ward: 1% of Sum Insured/day
- ICU/HDU: 2% of Sum Insured/day

**Step 2**: Compare the billed room rent per day to the policy cap.

**Step 3**: If billed > cap, compute `deduction_ratio = policy_cap / billed_rent`.

**Step 4**: Apply the deduction ratio proportionally to **all associated charges** — doctor fees, nursing, diagnostics, OT, anaesthesia. This is the critical point that hospitals most frequently get wrong: the deduction is not applied only to the room charge; it applies to all associated charges.

**Step 5**: Compute `cashlessApproved = totalBilled − totalDeductions − exclusions − copay`.

**Step 6**: Compute `patientShare = totalBilled − cashlessApproved`.

A daycare stay under 24 hours is exempt from room rent deduction review per IRDAI guidelines. A 12-hour or 18-hour observation stay must not trigger room rent queries.

---

### Q6. How should package validation work?
Package validation is **deterministic lookup** against a versioned package code table:

1. The procedure code (from the Procedure Coding Engine) is looked up against the insurer's package table.
2. If a package applies, itemized billing for the same procedure is flagged as a conflict.
3. Package rates are insurer-specific and scheme-specific (e.g., Star Health Medi-Classic has different package rates than Star Health Comprehensive).
4. Implant packages: the implant serial number must be present; implant cost must not exceed the package cap.
5. Hospital-defined rates: if a hospital has negotiated a rate above the standard package, the hospital rate takes precedence (uploaded via Hospital Admin API).
6. Package conflicts: if two mutually exclusive procedures are billed in the same episode, a `PACKAGE_CONFLICT` violation is raised.

---

### Q7. How should insurer-specific rules work?
Each insurer has a YAML policy file in `rules/insurers/{insurer_name}/`. The Orchestration Controller identifies the insurer from the TPR's policy number field, resolves the active scheme version by matching the admission date to the `effectiveFrom` / `effectiveTo` range, and loads the correct YAML file.

Insurer-specific overrides are a first-class concept in the YAML schema:
```yaml
insurerSpecificOverrides:
  - rule: "ICU_UPGRADE_REQUIRES_CLINICAL_JUSTIFICATION"
    value: true
  - rule: "DAYCARE_MINIMUM_HOURS"
    value: 0
```

New insurers are onboarded by:
1. Creating a new YAML file under `rules/insurers/`
2. Running a schema validation check
3. Deploying (hot-reload, no restart)

---

### Q8. How should hospital-specific packages work?
Hospitals upload their approved package rate cards via the **Hospital Admin API** (a separate administrative service, not part of Taiga). These rates are stored in a hospital package registry and loaded during rule resolution.

Hospital package overrides follow a strict rule: they can **only override the rate** of an approved insurer package. They cannot:
- Add new procedures to the package list
- Remove exclusions from the insurer's exclusion list
- Override waiting period rules

If a hospital-specific rate is higher than the insurer's package rate, the insurer's rate wins. If lower, the hospital's rate wins.

### Q9. How should explainability work?
Every output field in the FCA carries a `citations` array. Each citation follows the chain:

```
Policy Clause → Rule Applied → Calculation Performed → Financial Impact
```

Example (Room Rent):
```
"Room Rent exceeded policy limit by ₹1,500/day.
 Clause 4.2 — Proportional Deduction applies.
 Deduction Ratio: 3000 / 4500 = 0.667.
 Doctor Fee: ₹12,000 × 0.667 = ₹8,004 payable.
 Nursing: ₹6,000 × 0.667 = ₹4,002 payable.
 Total deduction: ₹5,994."
```

The explainability engine assembles these citations from the result objects produced by each engine. No citation is ever AI-generated — citations are templated strings with computed values inserted. This is critical: explainability cannot rely on an LLM to explain financial decisions.

### Q10. How should confidence be calculated?
Taiga's overall score (0–100) is a weighted composite:

| Dimension | Weight |
| :--- | :---: |
| ICD Validation | 15% |
| Procedure Validation | 15% |
| Policy Compliance | 25% |
| Room Rent Compliance | 20% |
| Billing Arithmetic | 15% |
| Package Compliance | 10% |

Each dimension scores 0–100. A blocking violation in any dimension automatically caps that dimension at 0. The overall score feeds into the `overallStatus`:
- ≥ 90: `PASS`
- 70–89: `CONDITIONAL`
- < 70: `FAIL`

Note: a `waitingPeriod = FAIL` or `exclusion = FAIL` forces `overallStatus = FAIL` regardless of the numeric score.

---

### Q11. When should human review be triggered?
Human review is triggered by **deterministic conditions only**:

| Condition | Trigger |
| :--- | :--- |
| ICD confidence < 0.85 | `humanReviewRequired = true` |
| Any arithmetic mismatch > ₹0 | `humanReviewRequired = true` |
| `waitingPeriod = FAIL` | `humanReviewRequired = true` |
| `exclusion = FAIL` | `humanReviewRequired = true` |
| `financialRisk = HIGH` | `humanReviewRequired = true` |
| `overallScore < 70` | `humanReviewRequired = true` |
| Policy YAML not loaded | `humanReviewRequired = true` |

Target: < 5% of cases require human review. For high-volume standard cases (Cataract, LSCS, Dengue) with complete documentation and standard room categories, the deterministic path should achieve `PASS` or `CONDITIONAL` with no human review needed.

---

### Q12. How should Taiga integrate with Claim Readiness?
Taiga's FCA is consumed by Claim Readiness as a **pre-condition gate**:

- `overallStatus = PASS` → Claim Readiness proceeds normally
- `overallStatus = CONDITIONAL` → Claim Readiness proceeds with advisory flags passed through
- `overallStatus = FAIL` → Claim Readiness **blocks submission** and routes to hospital billing for correction
- `financialRisk = HIGH` → Claim Readiness routes to senior human reviewer before proceeding

Claim Readiness does not re-run any Taiga calculations. It reads the FCA object and acts on it.

## 2. How Taiga Differs from Other Services

### Taiga vs Fairway

| Dimension | Fairway | Taiga |
| :--- | :--- | :--- |
| **Question answered** | Is clinical evidence sufficient? | Is the claim financially compliant? |
| **Primary domain** | Clinical medicine | Financial compliance & medical coding |
| **Evidence Layer** | Clinical facts from TPR | Financial facts from TPR + policy rules |
| **Reasoning Layer** | Clinical justification | Financial risk classification |
| **AI role** | Clinical narrative adequacy | Medical terminology normalization only |
| **Output** | CEA (MNS + ECS) | FCA (8 validation dimensions + risk) |
| **Deterministic guarantee** | Evidence discovery 100% deterministic | Financial calculations 100% deterministic |
| **Key metric** | Medical Necessity Score | Financial Compliance Score (0–100) |

Fairway and Taiga are **peers** in the pipeline. Taiga consumes Fairway's CEA as an input (not to re-evaluate clinical evidence, but to reference the necessity classification when making risk decisions). Taiga trusts Fairway's output completely — it never re-runs clinical checks.

---

### Taiga vs Claim Readiness

| Dimension | Taiga | Claim Readiness |
| :--- | :--- | :--- |
| **Purpose** | Validate compliance | Assess overall readiness to submit |
| **Scope** | Financial compliance only | Clinical + Financial + Administrative |
| **Inputs** | TPR + CEA | CEA + FCA + Admin data |
| **Runs calculations** | Yes — room rent, co-pay, deductions | No — reads results |
| **Output** | FCA | Claim Readiness Score |
| **Gate role** | Compliance gate | Submission gate |

Claim Readiness is a **consumer** of Taiga's output, not a peer. It aggregates CEA (from Fairway) + FCA (from Taiga) + administrative checks into a final submission decision.

---

### Taiga vs Aegis

| Dimension | Taiga | Aegis |
| :--- | :--- | :--- |
| **Timing** | Pre-submission | Post-denial |
| **Purpose** | Prevent denial | Reverse denial |
| **Input** | TPR + CEA | Denial letter + CEA + FCA |
| **Output** | FCA (compliance report) | Appeal letter + evidence package |
| **AI role** | Terminology normalization | Legal & medical appeal drafting |

Taiga eliminates the need for Aegis in the ideal case. If Taiga catches all compliance issues before submission, denials should be rare. Aegis is the safety net when insurers still deny despite a clean FCA.

## 3. Why Each Module Exists

| Module | Why It Exists |
| :--- | :--- |
| **ICD Coding Engine** | Wrong ICD codes are the #1 cause of claim rejection in India. Chapter locks prevent category errors. |
| **Procedure Coding Engine** | Incorrect CPT/procedure mapping leads to underpayment or package mismatch denials. |
| **Policy Rule Engine** | Every insurer has different waiting periods, exclusions, and daycare rules. They must be applied deterministically against the exact policy terms. |
| **Billing Validation Engine** | Arithmetic errors, duplicate charges, and non-payable consumables are common in hospital billing and create rejections. |
| **Room Rent Engine** | Room rent is India's most impactful proportional deduction category. Most hospitals get it wrong because they only deduct on room charges, not all associated charges. |
| **Package Validation Engine** | Package billing vs. itemized billing conflicts are a common denial reason, especially for Cataract, LSCS, and TKR. |
| **Financial Risk Engine** | Aggregates all violations into a single risk classification so downstream services have a clear decision signal. |
| **Explainability Engine** | Every financial decision must be justified with a policy clause and a calculation. No "black box" outputs. |

## 4. Deterministic vs AI — Decision Framework

The decision rule is simple:

> **If you can write the answer as a formula or a lookup, it must be deterministic.**
> **If the answer requires clinical or terminological knowledge, AI may assist.**

| Always Deterministic | AI Conditional |
| :--- | :--- |
| Room rent cap (1% / 2% of SI) | ICD code for ambiguous diagnosis |
| Waiting period date arithmetic | Procedure name normalization |
| Co-pay percentage calculation | Drug generic/brand resolution |
| Exclusion list matching | Diagnosis abbreviation expansion |
| Package rate lookup | |
| Arithmetic validation | |
| GST calculation | |
| Financial risk classification | |

**Why this boundary?** Because financial errors have direct monetary consequences for patients. A room rent deduction computed incorrectly by an LLM means a patient pays more than they should. A deterministic algorithm makes exactly the same calculation every time, is auditable, and is legally defensible.

## 5. Scalability

- **Stateless**: Each Taiga request is self-contained. No session state between requests.
- **Horizontal scaling**: Pod replicas behind a load balancer. Policy YAMLs loaded into shared Redis cache at startup.
- **Parallel execution**: 5 engines run concurrently in a single request, reducing wall-clock time from ~2000ms serial to ~400ms parallel.
- **Policy cache**: YAML files are loaded into memory at startup and refreshed on version change. Policy lookup latency is < 20ms.
- **Volume target**: Designed for 5,000+ concurrent claim validations.

---

## 6. GPU Usage

Taiga's target is **< 20% AI invocation rate**. This means in at least 80% of cases, no GPU is consumed at all.

GPU is consumed only when:
- An ICD code cannot be resolved deterministically (ambiguous diagnosis text)
- A procedure name is not found in the synonym dictionary
- A drug name requires generic/brand resolution

For standard high-volume Indian claim types (Cataract, LSCS, Dengue, Typhoid, Hypertension, Diabetes), the deterministic dictionary covers > 95% of cases without any LLM invocation.

---

## 7. Latency Optimization

| Technique | Impact |
| :--- | :--- |
| Parallel execution of 5 engines | Reduces wall-clock time from ~2000ms serial to ~400ms |
| Policy YAML cached in Redis | Policy load latency < 20ms |
| ICD dictionary loaded in memory | ICD lookup < 5ms |
| LLM invocation gated by confidence router | Invoked < 20% of cases |
| Package table indexed by procedure code | Package lookup O(1) |

The deterministic path (no LLM) consistently completes in < 730ms. The LLM path adds ~1200ms for terminology resolution, still within the 3000ms P95 target.
